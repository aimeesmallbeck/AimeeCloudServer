Platform Architecture

AimeeCloud uses a Distributed Agent Architecture - a hybrid architecture where the robot's local capabilities work seamlessly with cloud intelligence. The robot handles real-time tasks independently, while complex reasoning and learning happen in the cloud.

Layer Structure

AimeeCloud - Cloud AI (LLM, Memory, Learning)
🧠 Skill Layer - Domain-Specific Capabilities
ROS2 - Message Bus & State Management
🧱 Brick Layer - Arduino App Lab Bricks
📦 Node Layer - Individual ROS2 Nodes
Hardware - Motors, Sensors, Cameras

How It Works

When you speak to Aimee, your voice is captured on-device and sent to the cloud. The LLM processes your intent, determines what actions are needed, and sends commands back down the stack. Local execution happens on the UNO Q for real-time response, while complex reasoning lives in the cloud.

Neuro-Symbiotic Agent Structure

LLMs are surprisingly bad at managing and playing games. How can an agent that can code the most complex systems not even be able to play Tic-Tac-Toe? The answer lies in how they actually work.

This discrepancy highlights one of the most fundamental misunderstandings about how Large Language Models (LLMs) actually work. Language is a proxy for intelligence in humans, but it is an illusion of intelligence in AI.

To understand why an agent might struggle with Yahtzee despite having a great instruction manual, we have to look at the difference between Probabilistic Generation and Symbolic Logic.

The Improvisational Actor vs. The Accountant

Think of an LLM as a brilliant improvisational actor. If you ask it to act like a grandmaster chess player or a master coder, it will brilliantly sound like one. It uses its massive training data to probabilistically guess the most likely next word in a sequence. This makes it incredible at writing essays, translating languages, and writing code.

However, playing a game like Yahtzee requires an Accountant - a system that maintains absolute, rigid truth over time (State) and applies strict mathematical rules (Symbolic Logic).

Why the Improvisational Actor Fails at Being the Accountant

  • The Illusion of Memory: LLMs do not have internal variables. Their "memory" is just the chat history (Context Window). Every turn, the LLM has to re-read the entire conversation and probabilistically infer the current state. As the game goes on, it starts hallucinating the state.
  • Math is Not Language: LLMs do not calculate; they predict. If you roll [4, 4, 4, 2, 1], a traditional program sums this to 15 instantly. An LLM tries to guess what the next token should be based on similar text it has seen. It's remarkably bad at basic arithmetic.
  • Strategy Requires Forward Simulation: Grandmaster strategy requires branching logic. A traditional game AI uses algorithms like Minimax or Monte Carlo Tree Search to simulate hundreds of possible future rolls. An LLM operates linearly - it generates word by word, moving forward.

How to Fix the Architecture

We are fixing this by changing the architecture from a "Pure LLM" to a Neuro-Symbolic Agent. We separate the "Brain" from the "Scorekeeper":

  • The Game Engine (Python): Holds the absolute truth of the game (JSON state). Handles all math, validates legal moves, and tracks the score.
  • The LLM as Translator/Decider: Does not keep track of the game. Every turn, the engine injects the exact current JSON state into the prompt.
  • Structured Output: Aimee replies with structured commands (e.g., {"action": "hold", "dice": [0, 1]}), which the Game Engine executes and validates.

If Aimee tries to make an illegal move, the strict Game Engine rejects it, updates the prompt, and forces her to re-evaluate. This is the Neuro-Symbiotic Agent Structure - cloud intelligence with local game state truth.
